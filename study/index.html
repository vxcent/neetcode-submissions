Concurrency mental model

Get this straight and you'll pick the right tool instantly — and you'll have a crisp sentence to say out loud.

The decision that drives everything

The first question to ask (yourself and the interviewer): what is this actually waiting on?

The GIL, in one breath

Say this: "CPython's GIL lets only one thread execute Python bytecode at a time, so threads don't speed up CPU work — but the GIL is released during I/O, so threads (or asyncio) still give big wins for I/O-bound workloads. For CPU parallelism I'd reach for multiprocessing."

threads vs asyncio — same use case, different style

If the challenge hands you a blocking function (e.g. requests.get), threads/ThreadPoolExecutor are the low-risk choice. Reach for asyncio when the API is already async (aiohttp) or they ask for thousands of connections.

1 · Locks & shared state

I/O or CPUthreading.Lock

The foundation: any mutable state touched by multiple threads needs a lock. += is not atomic — it's read, add, write, and threads can interleave between those steps.

Without a lock, two "+1"s can both read 0 and both write 1 — you lose an increment.
import threading

class Counter:
    def __init__(self):
        self._value = 0
        self._lock = threading.Lock()

    def increment(self):
        with self._lock:          # read-add-write is now atomic
            self._value += 1

    @property
    def value(self):
        with self._lock:
            return self._value

What to know

  • Lock = mutual exclusion. RLock = re-entrant (same thread can acquire twice) — use when a locked method calls another locked method.
  • Always use with lock: so it releases even on exception.
  • Keep the critical section tiny — hold the lock only around the shared mutation, not the slow work.
  • A thread-safe cache is the same idea: guard the dict with a lock, or use one lock per shard for less contention.
Deadlock: two threads each holding a lock the other needs. Avoid by always acquiring multiple locks in a consistent order, or by not needing two locks (use a Queue instead).

2 · Producer / consumer + worker pool

I/O-boundqueue.Queue

The single most likely shape in a concurrency challenge. A thread-safe queue.Queue lets producers hand work to a fixed pool of worker threads — no manual locking needed, the queue handles it.

Producers put(), workers get(). The pool size caps concurrency; the queue decouples the two sides.
import threading, queue

def worker(q, results, lock):
    while True:
        item = q.get()
        if item is None:          # sentinel = shut down
            q.task_done()
            break
        out = item * item         # ... the real work ...
        with lock:
            results.append(out)
        q.task_done()

q = queue.Queue()
results, lock = [], threading.Lock()
workers = [threading.Thread(target=worker, args=(q, results, lock))
           for _ in range(4)]
for t in workers:
    t.start()

for i in range(20):               # produce
    q.put(i)

q.join()                          # block until every item is processed
for _ in workers:                 # one sentinel per worker
    q.put(None)
for t in workers:
    t.join()

The three things that trip people up

  • Shutdown: workers loop forever — send a None sentinel per worker (or use a shared "stop" Event).
  • task_done / join: call q.task_done() for every get(); q.join() waits until the count hits zero.
  • Results: if workers append to a shared list, guard it — or have each return via a results Queue.
Bounded queue: queue.Queue(maxsize=100) gives you backpressure — producers block when it's full, so a fast producer can't exhaust memory.

3 · Bounded concurrency / rate limiter

I/O-boundthreading.Semaphore

"At most N at once." A Semaphore is a counter with a floor of zero — acquire() blocks when N are already in flight. Perfect for "don't hammer the API with more than 5 concurrent requests."

import threading

sem = threading.Semaphore(3)      # at most 3 concurrent

def fetch(url):
    with sem:                     # blocks if 3 are already running
        return do_request(url)    # ... work ...

threads = [threading.Thread(target=fetch, args=(u,)) for u in urls]
for t in threads: t.start()
for t in threads: t.join()

Rate limiting flavors — name them

  • Concurrency limit (above): N in flight at once → Semaphore.
  • Requests-per-second: token bucket — refill tokens at a fixed rate, each call spends one, block/reject when empty. Mention this if they ask for "10 req/sec."
  • Async twin: asyncio.Semaphore(N) with async with.

4 · Fan-out / fan-in

I/O-boundconcurrent.futures

The cleanest API under time pressure. Submit many jobs to a ThreadPoolExecutor, collect results as they finish. Reach for this first when the task is "do these N independent things in parallel."

from concurrent.futures import ThreadPoolExecutor, as_completed

def fetch(url):
    return len(do_request(url))    # ... work, returns a result ...

urls = [...]
with ThreadPoolExecutor(max_workers=8) as pool:
    future_to_url = {pool.submit(fetch, u): u for u in urls}
    for fut in as_completed(future_to_url):
        url = future_to_url[fut]
        try:
            size = fut.result()    # re-raises worker exceptions here
        except Exception as e:
            print(f"{url} failed: {e}")
        else:
            print(f"{url} -> {size}")

Why interviewers like this

  • with ... as pool: handles startup/shutdown for you — no manual thread juggling.
  • pool.map(fn, iterable) is even terser when you want results in order and don't need per-item error handling.
  • as_completed gives results as they finish — good for streaming / progress.
  • Exceptions are captured in the Future and re-raised at .result() — handle them there.
  • Swap to ProcessPoolExecutor (same API!) if it turns out CPU-bound.

5 · asyncio

I/O-bound, high fan-outasync / await

One thread, an event loop, cooperative multitasking. While one coroutine awaits I/O, the loop runs others. Scales to thousands of concurrent connections cheaply.

Only one coroutine runs at a time; await is where it hands control back to the loop.
import asyncio

async def fetch(url, sem):
    async with sem:                       # bound concurrency
        await asyncio.sleep(0.1)          # stand-in for real async I/O
        return url.upper()

async def main(urls):
    sem = asyncio.Semaphore(8)
    tasks = [fetch(u, sem) for u in urls]
    return await asyncio.gather(*tasks, return_exceptions=True)

results = asyncio.run(main(urls))

The essentials

  • asyncio.run(main()) is the single entry point; await only works inside async def.
  • asyncio.gather(*tasks) = fan-out + wait-all. return_exceptions=True stops one failure from cancelling the rest.
  • asyncio.Queue, asyncio.Semaphore, asyncio.Lock mirror the threading versions with await.
  • Never call a blocking function directly — it freezes the whole loop. Wrap it: await asyncio.to_thread(blocking_fn, arg).
  • Cancel/timeout: await asyncio.wait_for(coro, timeout=5).
If you're unsure between threads and asyncio in the live round and the given I/O is blocking, use threads — it's fewer moving parts. Bring up asyncio as "how I'd scale this to thousands of connections."

System design: the framework

You know the pieces — what wins is a repeatable order so you never freeze on a blank whiteboard. Same 7 steps every time.

Step 1 is where mid-level candidates win or lose

Spend the first 5–8 minutes driving requirements, not drawing. Nail down: who uses it, the 2–3 core features (cut the rest), rough scale, and which of consistency vs availability vs latency matters most. Write them in a corner of the board and refer back — every later decision cites one of them.

Estimation, the lazy way

  • Pick round numbers: "say 1M daily users, 10 actions each → 10M/day ≈ ~120 writes/sec average, ~10× peak → ~1200/sec."
  • Reads usually dominate writes (often 100:1) → that's your cue for caching + read replicas.
  • Storage: bytes/record × records/day × retention. One or two significant figures is plenty.
Say your assumptions out loud and let the interviewer correct them — that's collaboration, and it de-risks the whole design.

Building blocks

The vocabulary you assemble every design from. Know what each does, where it goes, and its main tradeoff.

The default skeleton — start here, then justify each box from your requirements.

Load balancer

Spreads traffic across app servers; enables horizontal scaling + failover. Round-robin / least-connections. Health checks remove dead nodes.

Cache

Redis/Memcached in front of the DB for hot reads. Decide: what to cache, TTL/eviction (LRU), and invalidation (the hard part). Cache-aside is the common pattern.

SQL vs NoSQL

SQL: relations, transactions, strong consistency. NoSQL: flexible schema, horizontal scale, tuned to an access pattern. Pick from your data model, not hype.

Indexing

Speeds reads on queried columns, slows writes and costs space. Index what you filter/sort on.

Sharding

Partition data across nodes by a shard key (e.g. user_id) when one DB can't hold the load. Tradeoff: cross-shard queries get painful; pick the key by access pattern.

Replication

Copies for read scaling + failover. Primary-replica: writes to primary, reads from replicas → replication lag = eventual consistency on reads.

Message queue

Kafka/SQS/RabbitMQ decouple producers from consumers, absorb spikes, enable async work (emails, feeds). Adds eventual consistency + at-least-once delivery to reason about.

CDN

Caches static/media at edge locations near users → lower latency, less origin load.

CAP, in plain words

When the network splits, you either refuse some requests to stay consistent (CP) or answer with possibly-stale data to stay available (AP). Say which one your system chooses and why — e.g. "a bank ledger is CP; a social feed is AP."

Both rounds are "collaborative"

The word appears in both descriptions on purpose. They're testing how you work with someone, not just what you know.

In the coding round

  • Clarify before coding: "I/O or CPU-bound? How many tasks? Any ordering guarantee?"
  • Narrate as you type: "I'll use a thread pool here so I don't hand-manage threads…"
  • Start simple (a working serial version), then make it concurrent — a running solution beats a perfect unfinished one.
  • When stuck, think out loud and invite input rather than going silent.

In the design round

  • Drive requirements first — treat the interviewer as your PM, confirm scope and cut features.
  • State assumptions and let them steer: "I'll assume read-heavy — correct me if not."
  • Draw boxes and arrows fast and legibly; label the data flow. Practice on Excalidraw (free, no login).
  • Volunteer tradeoffs before they ask — "the cost here is replication lag" — that reads as senior thinking.
Golden rule: a communicated good answer scores higher than a silent great one. Talk.
